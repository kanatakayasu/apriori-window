以下は、あなたが貼ってくれた 資料1（Claude全体像：Project×Skills×MCP×Claude Code…） と 資料2（Skillを“自動最適化”する考え方） をそのまま設計原則に落として、skills/ と claude.md（= Claude Code運用のルール・入口ドキュメント）を作るための具体手順・テンプレです。

⸻

1) まず整理：skills と claude.md の役割分担

skills（= タスク手順書の“部品”）
	•	目的：特定タスクを「毎回同じ品質で」実行するための手順を固定化する（資料1の “マクロ” 的な使い方）。
	•	置き場：skills/<skill_name>.md
	•	中身：入力→処理→出力→品質チェックまで、実行可能な手順として書く（資料2で言う「どうやるか」を書く）。

claude.md（= リポジトリ全体の“運用憲法”）
	•	目的：Claude Code（やCowork的な実行環境）に、このリポジトリでの開発・編集の“やり方”を教える。
	•	置き場：リポジトリ直下 claude.md
	•	中身：作業ポリシー、ディレクトリ構造、実行コマンド、テスト、PR流儀、禁止事項、出力形式など。

ざっくり：
claude.md は 全タスク共通のガイドライン、skills/*.md は タスクごとの手順書。

⸻

2) skills の設計原則（資料2の「自動改善できるSkill」に寄せる）

資料2の肝は、「Skillを“長い構造化ドキュメント”として扱い、成果物と正解の差分からSkillを改善する」でした。
これを前提にすると、skills は次の条件を満たすと強いです。

skills に必ず入れるべきブロック（重要）
	1.	Purpose（このSkillが何を達成するか）
	2.	Inputs（入力の形式・必須項目・不足時の扱い）
	3.	Outputs（成果物の形式・章立て・ファイル名規約）
	4.	Procedure（手順：再現可能なステップ）
	5.	Quality Gates（品質チェック：整合性・網羅性・数値矛盾など）
	6.	Failure Modes（よくある失敗と回避策）
	7.	Evaluation（評価方法：スコアリング観点やチェックリスト）
	8.	Examples（入力例→出力例の骨子）

資料2で起きた改善（ROI計算の具体化、周辺コスト、整合性確認、案件固有ロジック）って、ぜんぶ Quality Gates と Failure Modes に落ちます。ここが薄いSkillは育ちにくいです。

⸻

4) 「良いSkill」になる書き方（資料2の学びを反映）

A. “含める/示す”だけで終わらせない

資料2の初期Skillが弱かった理由はこれです。
動作するSkillは、最低限「どう計算するか / どう検証するか」まで書きます。

例（ROIを含める、を育てる）：
	•	NG：ROI試算を含める
	•	OK：ROIは、(1)コスト分解→(2)便益分解→(3)回収期間算出→(4)サマリーと一致確認 の順で作る。周辺コスト（運用/外部委託等）を案件カテゴリ別に必ず加える。

B. Quality Gates を“強く”するほど自動最適化が回りやすい

資料2の「矛盾（14ヶ月 vs 7.6年）」みたいな事故は、品質ゲートで検出できます。
Skillに「矛盾検出の手順」を書くと、次の生成から安定します。
	•	サマリーの数値が本文の計算結果と一致しているか
	•	前提（単位、期間、通貨）が全章で一貫しているか
	•	比較軸が案件カテゴリに必要十分か（セキュリティならMDR等）

C. “案件タイプ別の分岐”は Skill に書く

資料2の「セキュリティ案件固有コスト」みたいなものは、手順の中に分岐を持たせます。

例：
	•	案件タイプ=セキュリティの場合：ライセンス以外に運用監視(MDR/SOC)・インシデント対応工数・監査対応などを見積項目に追加
	•	案件タイプ=会計の場合：月次締め短縮、監査対応、仕訳自動化などを便益分解

⸻

5) Skills を“自動最適化”する運用（資料2を実務に落とす）

リポジトリに、以下の3点セットを置くと回り始めます。

(1) ゴールデン（正解データ）置き場
	•	eval/goldens/<task_id>/final_output.*
	•	eval/inputs/<task_id>/context.*

(2) 評価プロンプト（採点基準）
	•	eval/rubric.md（6軸×0-100など、資料2の方式）
	•	重大欠陥（矛盾、捏造、必須項目欠落）は即NG

(3) ループ手順（改善の型）
	•	eval/run.md に、毎回の回し方を書く

運用の流れ（資料2の①〜④をそのまま）：
	1.	初期Skillで生成（goldenは見ない）
	2.	生成物 vs golden を比較
	3.	差分を “テキスト勾配” として記述（どこが/なぜ/どう直す）
	4.	Skillを改訂
	5.	複数案件で再評価（過学習を監視：資料2の4回目退行の対策）

⸻

6) claude.md の作り方（Claude Codeに“このリポでのやり方”を教える）

claude.md は、Claude Codeが迷わず動けるように、次を固定化すると強いです。

claude.md に入れるべき章
	1.	このリポの目的 / 成果物の定義
	2.	ディレクトリ構造（どこに何があるか）
	3.	作業原則（捏造禁止、変更範囲、最小差分、ログの残し方）
	4.	コマンド集（lint/test/build/run）
	5.	ブランチ/PR流儀（あなたはGit運用もしてるので相性良い）
	6.	Skills の参照方法（どのSkillをいつ使うか）
	7.	Evaluation（golden評価の回し方）

⸻

7) claude.md テンプレ（そのまま置ける）

# claude.md

## 0. Repository Purpose
- このリポジトリの目的：
- 主要な成果物：
- 重要な品質基準：

## 1. Directory Layout
- /skills : タスク手順書（Skill）
- /eval   : golden評価（inputs, goldens, rubric, run手順）
- /src    :
- /docs   :

## 2. Working Principles
- 事実不明な点は推測しない。必要なら質問する/「不明」とする。
- 変更は最小差分で行い、意図を説明する。
- 生成物は必ずQuality Gatesを自己チェックしてから提出する。
- 機密情報・認証情報（APIキー等）をコミットしない。

## 3. Default Output Format
- 文書：Markdown優先（必要ならdocx/pptx）
- 表：CSV or Markdown table
- コード：既存のlint/testに従う

## 4. Commands
- Install:
- Lint:
- Test:
- Run:
（このリポで使うコマンドを列挙）

## 5. Git / PR Workflow
- 作業ブランチ：feature/<name>
- コミット粒度：意味単位（1コミット1意図）
- PRに必ず書くこと：変更点 / 動作確認 / 影響範囲 / スクショ（任意）

## 6. Skills Usage
- 文章生成系：skills/<...>.md を優先参照
- データ整理系：skills/<...>.md
- 迷ったら：目的に最も近いSkillを選び、Inputs不足は質問する

## 7. Evaluation Loop (Skill Optimization)
- eval/inputs から入力を取り、Skillで生成（goldensは参照しない）
- eval/rubric.md の観点で採点
- 差分（テキスト勾配）を記録：eval/logs/<date>_<skill>.md
- Skillを更新し、複数タスクで再評価して過学習を避ける


⸻

8) 最短の作成手順（今日から使える）
	1.	claude.md を上のテンプレで作る（まず空でも良いので“入口”を用意）
	2.	よくある作業を3つ選んで skills/ に作る
	•	例：議事録→Word化、要件整理、提案書ドラフト、リサーチ要約…
	3.	それぞれに Quality Gates を最低5個入れる（数値整合・根拠・網羅・禁止事項など）
	4.	eval/ に “過去の完成品（golden）” を2〜4件置く（資料2の条件）
	5.	2〜3イテレーションだけ回して、Skillを育てる（資料2でも改善の大半は最初の2回）
