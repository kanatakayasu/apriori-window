1. 従来手法

従来手法：Apriori ＋ スライディングウィンドウ

アルゴリズムの流れ

  • トランザクション列を時系列に並べる
  • 幅 W のウィンドウをスライド
  • ウィンドウ内で Apriori を適用し頻出パターンを検出
  • 支持度 ≥ min_sup の区間を「密集区間」として記録
定義：密集区間

アイテムセット S が時刻区間 (ts, te) において
支持度 ≥ min_sup を満たす最大区間

出力

各アイテムセット → 密集区間のリスト
例: {A,B} → [(10, 45), (80, 120)]

1. 従来手法の課題

課題①：バスケット構造の無視による偽共起

実際の購買データでは、1トランザクションが
複数の独立した購入単位（バスケット）を含む。

トランザクション 0: basket 0: { A, B } ← 食料品 basket 1: { C, D } ← 日用品
問題

従来手法は「同一トランザクション内 = 共起」と判定するため、
A と C、A と D など別バスケット間の偽共起が発生する。

具体例

ペア実態従来
{A, B}同一basket○
{C, D}同一basket○
{A, C}別basket○（偽）
{B, D}別basket○（偽）

1. 従来手法の課題

課題②：密集区間の説明不足

密集区間は「いつ何が集中購入されたか」を示すが、
「なぜその時期に集中したか」を外部から説明できない。

問題

セール・キャンペーン等の外部イベントが密集区間と
どのような時間的関係にあるかが不明のまま。

必要な情報

  • 密集区間の開始・終了タイムスタンプ
  • 外部イベントの開始・終了タイムスタンプ
  • 両者の時間的関係の分類

期待する出力例

{A,B}: (10, 45) ← EventFollowsDense ← sale_event: (46, 60)

提案手法

提案手法の概要

Phase 1 の解決策

共起定義を
「同一バスケット内での共起」に変更。
basket_id の積集合が空でないペアのみを評価する。

Phase 2 の解決策

外部イベントリスト（JSON）と密集区間を照合し、
6種の時間的関係を列挙して出力する。

実装

Python（プロトタイプ）→ Rust（高速化） の二段構成。
Rust では rayon 並列処理・二分探索・BufWriter を活用。

2. Phase 1

Phase 1：バスケット粒度の共起定義

basket_id マップの構築

item → [basket_id, ...] (item_basket_map)
basket_id → transaction_id (basket_to_tx)

multi-item の共起カウント

  1. 各アイテムの basket_id リストを取得
  2. basket_id リストの積集合を取る
  3. 積集合を transaction_id に変換（重複を保持）
  4. transaction_id 列に対して密集区間を計算
効果

別バスケットのアイテム間は basket_id 積集合が空 → 自動的に排除

basket 0: {A,B} basket 1: {C,D} ibm[A] = [0], ibm[C] = [1] ∩ = [] (空) → 偽共起なし

find_dense_itemsets のアルゴリズム

3. スタックケース

スタックケースとは何か

タイムスタンプに重複がある場合、密集区間の左端 l が進まず
無限ループに陥る可能性がある。

発生条件

条件

ウィンドウ内カウント count ≥ threshold かつ
surplus = count - threshold 番目の値が
現在の左端 l と同一タイムスタンプ

なぜループするか

通常: l = ts[surplus] で左端を進める
問題: ts[surplus] == l なら左端が動かない → ループ

具体例

ts = [5,5,5,5,5, 8, 12] window_size = 10, threshold = 3 l=5: count=7, surplus=4 ts[4]=5 == l=5 → 動かない! → 無限ループ
9
3. スタックケース

スタックケース：修正手法

修正: フォールバック条件
if window_occurrences[surplus] == l {
    l += 1;  // 強制的に1進める
} else {
    l = window_occurrences[surplus];
}

window_occurrences[surplus] == l のとき、
l += 1 で強制的に左端を進める。
これにより必ず終了が保証される。

修正後の動作

ts = [5,5,5,5,5, 8, 12] window_size=10, threshold=3 l=5: surplus=4, ts[4]=5==l → l += 1 → l=6 l=6: window=[6,16], count=2 → count < threshold → 終了
密集区間の記録タイミング

count ≥ threshold を最後に満たした
左端・右端を記録してから終了

3. スタックケース

スタックケース後も密集が継続する場合

フォールバック後、右端から新たなトランザクションが入り直して
支持度が回復するケースを正しく処理する必要がある。

ts = [5,5,5,5,5, 8, 12, 16] window_size=10, threshold=3 l=5: surplus=4, ts[4]=5==l → l=6 (フォールバック) l=6: window=[6,16], count=3 → count == threshold → 継続! l=9: count=2 < threshold → 終了 記録: (5, 16) ← 正しい区間
ポイント

l のループ中、密集条件を満たすたびに
(start, end) を更新し続ける。
終了時に最後に更新した値を出力。

テストケース

ts=[5,5,5,5,5,8,12,16]
→ (5, 16) が返ることを確認

4. Phase 2

Phase 2：外部イベントとの時間的関連付け

入力

  • Phase 1 の出力: 密集区間 I = (ts,i, te,i)
  • 外部イベントリスト（JSON）: J = (ts,j, te,j)
  • 許容誤差 ε、最小重複長 d0

出力

  • アイテムセット × 密集区間 × イベント × 関係名
  • CSV 形式で出力
6種の時間的関係

DFE: 密集の直後にイベント開始
EFD: イベントの直後に密集開始
DCE: 密集がイベントを包含
ECD: イベントが密集を包含
DOE: 密集がイベントに先行重複
EOD: イベントが密集に先行重複

4. Phase 2

6種の時間的関係：定義

関係名略称条件
DenseFollowsEventDFE te_i − ε ≤ ts_j ≤ te_i + ε
EventFollowsDenseEFD te_j − ε ≤ ts_i ≤ te_j + ε
DenseContainsEventDCE ts_i ≤ ts_j ∧ te_i + ε ≥ te_j
EventContainsDenseECD ts_j ≤ ts_i ∧ te_j + ε ≥ te_i
DenseOverlapsEventDOE ts_i < ts_j ∧ te_i − ts_j ≥ d₀ − ε ∧ te_i < te_j + ε
EventOverlapsDenseEOD ts_j < ts_i ∧ te_j − ts_i ≥ d₀ − ε ∧ te_j < te_i + ε

I = 密集区間 (ts_i, te_i)，J = イベント (ts_j, te_j)，ε = 許容誤差，d₀ = 最小重複長

関連研究

手法解決する問題本研究との差異

本研究 Phase 1 偽共起の排除 basket_id 積集合による厳密な共起定義
