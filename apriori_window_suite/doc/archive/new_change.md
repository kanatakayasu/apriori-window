## 現状のアルゴリズム概要

- **入力**: トランザクション列（各行＝1トランザクション＝スペース区切りアイテムID）
- **処理**: Apriori + スライディングウィンドウで、アイテムセットが密集する時間区間を検出
- **出力**: `{アイテムセット} → 密集区間リスト [(start, end), ...]`

主要な関数（`main.rs` / `apriori_window_pruned.py`）:
- `compute_item_timestamps_map`: アイテムが出現したトランザクションIDのリストを作成
- `compute_dense_intervals`: スライディングウィンドウで密集区間を計算
- `find_dense_itemsets`: Aprioriで全頻出アイテムセットの密集区間を探索
  - アイテムセットのタイムスタンプ = 各アイテムのタイムスタンプリストの**積集合**（同一トランザクションに全アイテムが出現した時刻）

## 課題①：トランザクション内のバスケット構造の無視

### 問題

- **現状**: 1トランザクション = 1アイテム集合（全アイテムが同時購入とみなす）
- **実態**: 1トランザクション内に複数の独立したバスケット（購入単位）が混在する可能性がある
- **影響**: 異なるバスケットのアイテム間に**偽の共起関係**が生まれる

### 具体例

同一トランザクションで A, B, C が購入されている時:

| パターン | 解釈 | A-B共起 | B-C共起 | A-C共起 |
|---------|------|---------|---------|---------|
| 従来: `{A, B, C}` | 全て同時購入 | ✓ | ✓ | ✓ |
| 新手法: `{A, B} \| {C}` | 別々の購入単位 | ✓ | ✗ | ✗ |
| 新手法: `{A} \| {B, C}` | 別々の購入単位 | ✗ | ✓ | ✗ |

### アルゴリズムへの影響（概念）

**現状の共起定義**:
- アイテムセット{A, B}のタイムスタンプ = 「Aが出現したトランザクションID」∩「Bが出現したトランザクションID」
- 同一トランザクションにAとBが（どのバスケットであれ）出現していれば共起とみなす

**新手法の共起定義**:
- アイテムセット{A, B}のタイムスタンプ = 「AとBが**同一バスケット内で**共起したトランザクションID」のリスト（重複あり）
- 同一トランザクションでも別バスケットなら共起しない
- 1トランザクション内の複数バスケットに共起すれば、そのトランザクションIDが複数回出現する（バスケット粒度の計数）

**`compute_dense_intervals` への波及**:
- 共起タイムスタンプに重複が生じるため、ストライド調整にスタックケースの処理を追加する必要がある
- スタックケース（`window_occurrences[surplus] == l`）では `l += 1` にフォールバック
- 詳細は [phase1_impl_plan.md](../phase1_impl_plan.md) を参照

## 課題②：密集区間と外部イベントの関連付け

### 問題

- **現状の出力**: 「ある時間帯にこのアイテム集合が密集していた」のみ
- **欲しい出力**: 「その密集は○○イベントと△△の時間的関係にある」

### イベントの時間的関係

（`IMG_7412.jpg` / Table 2: Temporal Relations between Events より）

以下、密集区間を `e_i = [ts_i, te_i]`、外部イベントを `e_j = [ts_j, te_j]` とし、ε は許容誤差。

| 関係 | 記法 | 条件 |
|------|------|------|
| **Follows** | e_i → e_j | `te_i ± ε ≤ ts_j`（e_i が終わった直後に e_j が始まる） |
| **Contains** | e_i ≥ e_j | `(ts_i ≤ ts_j) ∧ (te_i ± ε ≥ te_j)`（e_i が e_j を包含する） |
| **Overlaps** | e_i ⊙ e_j | `(ts_i < ts_j) ∧ (te_i ± ε < te_j) ∧ (te_i − ts_j ≥ d_0 ± ε)`（部分的に重なる） |

> `d_0` は最小オーバーラップ長（アプリケーション依存のパラメータ）

### 具体例

| ドメイン | イベント | 関係種別 | 密集区間のアイテム |
|---------|---------|---------|----------------|
| ECサイト | アイドルのライブ | Follows（ライブ**直前**に購買密集） | アイドルグッズ |
| コンビニ | レジ横にガム配置 | Contains（配置**期間中**に密集） | ガム |
| スーパー | 特売チラシ配布 | Overlaps（チラシ後に密集が始まる） | 特売品 |

### 有意性の評価（Future Work）

関係が偶発的でないことの判定方法の候補:

- **選択肢A**: 全期間に対する密集区間の比率 vs イベントとの重複率を比較（シンプル）
- **選択肢B**: 統計検定（カイ二乗検定など）
- **選択肢C**: ランダムイベントとの比較（permutation test）

→ まずはブルートフォースで全関係を列挙することを優先する。
