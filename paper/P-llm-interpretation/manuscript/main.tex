\begin{abstract}
頻出アイテムセットマイニングは小売・Web ログ等の大規模トランザクションデータから有用なパターンを発見する基盤技術であるが，
検出されたパターンの解釈はドメイン専門家に依存しており，分析のボトルネックとなっている．
本論文では，密集アイテムセット（一定の時間ウィンドウ内で共起頻度が閾値を超えるアイテムの組み合わせ）の
検出結果を大規模言語モデル（LLM）に入力し，自然言語による解釈を自動生成するフレームワークを提案する．
提案手法は3つの主要コンポーネントから構成される：
(1) パターン構造の数値的特徴を抽出するパターン情報抽出器，
(2) 構造化プロンプトテンプレートによるLLMへの入力生成器，
(3) 解釈品質を定量的に評価するメトリクス体系．
実験では，モック LLM バックエンドを用いたプロトタイプにより，
パターンの時間的特徴（単一連続・断続的・高頻度断続）を自動分類し，
ビジネス仮説を含む解釈を100\%の生成率で提供できることを示す．
提案フレームワークは LLM バックエンドに非依存な設計であり，
任意の LLM API に接続可能なデモシステムとして実装した．
\end{abstract}

\section{はじめに}
\label{sec:intro}

頻出アイテムセットマイニング~\cite{agrawal1994fast, han2000mining}は，
トランザクションデータベースから高頻度で共起するアイテムの組み合わせを発見する
データマイニングの基本的なタスクである．
近年，スライディングウィンドウに基づく密集区間検出手法が提案され，
パターンの時間的局在性を捉える能力が向上している．

しかし，検出されたパターンの\emph{解釈}は依然として人手に依存している．
例えば，「アイテム$\{1, 2, 3\}$がトランザクション$[50, 120]$で密集」
という検出結果を見ても，なぜこのパターンが生じたのか，
ビジネス的にどのような意味を持つのかは自明ではない．

大規模言語モデル（LLM）~\cite{brown2020gpt3, openai2023gpt4}の
急速な発展により，構造化データの自然言語説明の自動生成が
現実的な技術となっている~\cite{zhang2023llm4dm}．
LLMは few-shot 学習~\cite{brown2020gpt3}や
Chain-of-Thought 推論~\cite{wei2022chain}によって，
複雑な分析タスクにも対応可能であることが示されている．

本論文では，密集アイテムセットパターンの検出結果をLLMに入力し，
自然言語による解釈を自動生成するフレームワークを提案する．
主要な貢献は以下の通りである：

\begin{itemize}
    \item パターンの時間的構造（区間数・カバー率・ギャップ幅）を
          数値的特徴として抽出し，LLMが解釈しやすい形式に変換する
          パターン情報抽出器の設計
    \item 単一パターン解釈・複数パターン比較・バッチ要約の3種類の
          プロンプトテンプレートの体系的設計
    \item 解釈の完全性（時間的記述・仮説生成の有無）と
          一貫性（カバー率と確信度の相関）を定量化する評価指標
    \item LLMバックエンドに非依存なプラグイン設計による
          デモシステムの実装
\end{itemize}

\section{関連研究}
\label{sec:related}

\subsection{頻出パターンマイニング}

Apriori~\cite{agrawal1994fast}は反単調性を利用した候補生成・剪定による
頻出アイテムセット発見の古典的アルゴリズムである．
FP-Growth~\cite{han2000mining}は候補生成なしのアプローチを提案した．
時間的側面については，エピソードマイニング~\cite{mannila1997discovery}が
イベント列からの頻出パターン発見を扱い，
統計的有意性の評価~\cite{webb2007discovering}も研究されている．

密集区間マイニングは，スライディングウィンドウ内のサポートが
閾値を超える連続的な時間区間を特定する手法であり，
パターンの時間的局在性を明示的に捉える点で従来の頻出パターンマイニングと異なる．

\subsection{LLM によるデータ分析支援}

GPT-3~\cite{brown2020gpt3}以降，LLMのデータ分析への応用が
活発に研究されている~\cite{zhang2023llm4dm}．
GPT-4~\cite{openai2023gpt4}は複雑な推論タスクにおいても高い性能を示し，
Chain-of-Thought プロンプティング~\cite{wei2022chain}は
段階的推論による解釈生成を可能にした．
テキスト要約~\cite{narayan2018dont}の技術もパターン解釈の基盤となる．

しかし，マイニング結果の自然言語解釈に特化したフレームワークは
まだ確立されておらず，本研究はこのギャップを埋めるものである．

\section{問題定義}
\label{sec:problem}

\begin{definition}[密集アイテムセット]
\label{def:dense}
トランザクションデータベース$\mathcal{D} = \{T_0, T_1, \ldots, T_{n-1}\}$，
ウィンドウサイズ$W$，閾値$\theta$が与えられたとき，
アイテムセット$I \subseteq \mathcal{I}$の\emph{密集区間}$[s, e]$とは，
すべての$l \in [s, e]$について$I$のウィンドウ$[l, l+W]$内での
共起回数が$\theta$以上となる極大連続区間である．
\end{definition}

\begin{definition}[パターン解釈タスク]
\label{def:interpretation}
密集アイテムセット$(I, \{[s_1, e_1], \ldots, [s_k, e_k]\})$と
オプショナルなアイテムメタデータ$M: \mathcal{I} \to \text{String}$が
与えられたとき，パターン解釈タスクとは，以下を含む自然言語テキストを
生成することである：
\begin{enumerate}
    \item \textbf{要約}: パターンの1文要約
    \item \textbf{時間的記述}: 密集区間の時間的特徴の説明
    \item \textbf{ビジネス仮説}: パターンが生じた原因の仮説
\end{enumerate}
\end{definition}

\begin{definition}[パターン特徴量]
\label{def:features}
パターン$(I, \mathcal{S})$（$\mathcal{S} = \{[s_j, e_j]\}_{j=1}^{k}$）に対して，
以下の特徴量を定義する：
\begin{itemize}
    \item カバー率: $\rho = \sum_{j=1}^{k}(e_j - s_j + 1) / n$
    \item 区間数: $k = |\mathcal{S}|$
    \item 最大ギャップ: $g_{\max} = \max_{j=1}^{k-1}(s_{j+1} - e_j)$
    \item 時間パターン分類:
        $\tau = \begin{cases}
            \text{単一連続} & k = 1 \\
            \text{断続的}   & 2 \leq k \leq 3 \\
            \text{高頻度断続} & k \geq 4
        \end{cases}$
\end{itemize}
\end{definition}

\begin{theorem}[解釈完全性]
\label{thm:completeness}
パターン特徴量$(\rho, k, g_{\max}, \tau)$とメタデータ$M$が
与えられたとき，定義~\ref{def:interpretation}の3つの解釈要素
（要約・時間的記述・ビジネス仮説）はすべて生成可能である．
\end{theorem}

\begin{proof}
各要素の生成可能性を示す．
(1) 要約は$\tau$と$\rho$から時間パターン分類と強度を記述することで生成できる．
(2) 時間的記述は$k$, $g_{\max}$, 各$[s_j, e_j]$から区間の分布を記述できる．
(3) ビジネス仮説は$\rho$の大きさに応じた定型パターン
（強パターン：定番商品，中パターン：季節商品，弱パターン：ニッチ商品）
とメタデータ$M$のカテゴリ情報を組み合わせて生成できる．
\end{proof}

\section{提案手法}
\label{sec:method}

提案フレームワークは3つのコンポーネントから構成される（図~\ref{fig:architecture}）．

\begin{figure}[t]
    \centering
    \fbox{\parbox{0.9\linewidth}{
        \centering
        \textbf{システムアーキテクチャ}\\[5pt]
        密集パターン検出 $\to$ 特徴量抽出 $\to$ プロンプト生成 $\to$ LLM $\to$ 解釈出力\\[5pt]
        \small{(1) PatternInfo抽出器 \quad (2) PromptBuilder \quad (3) LLMバックエンド}
    }}
    \caption{提案フレームワークのアーキテクチャ．密集パターン検出の出力を
    構造化特徴量に変換し，テンプレートベースのプロンプトを生成して
    LLMに入力する．}
    \label{fig:architecture}
\end{figure}

\subsection{パターン情報抽出器}

密集アイテムセットマイニングの出力$\{(I_i, \mathcal{S}_i)\}_{i=1}^{m}$から，
各パターンの構造化特徴量を抽出する．
具体的には，定義~\ref{def:features}に基づき，
カバー率$\rho$，区間数$k$，最大ギャップ$g_{\max}$，
時間パターン分類$\tau$を計算する．

\subsection{プロンプトテンプレート}

3種類のプロンプトテンプレートを設計した：

\subsubsection{単一パターン解釈プロンプト}
1つのパターンに対して，アイテムセット，密集区間，カバー率等の
数値的特徴を構造化して LLM に入力する．
メタデータ（商品名・カテゴリ）が利用可能な場合は追加情報として含める．

\subsubsection{比較分析プロンプト}
複数のパターンを同時に入力し，時間的重なり，共通アイテム，
ビジネス的関連性の観点から比較分析を求める．

\subsubsection{バッチ要約プロンプト}
全検出パターンの一括要約を求める．
各パターンの1文要約と全体傾向の記述を含む．

\subsection{LLM バックエンド}

フレームワークはバックエンドに非依存な設計とし，
\texttt{PatternInterpreter}クラスの\texttt{llm\_backend}パラメータで
切り替え可能とした．
本論文ではデモ用のモックバックエンドを実装し，
ルールベースの応答生成によりフレームワークの動作を検証する．

モックバックエンドは以下のルールに基づいて応答を生成する：
\begin{itemize}
    \item 時間パターン分類$\tau$に応じた時間的記述の生成
    \item カバー率$\rho$に応じた確信度の設定
        （$\rho > 0.3$: 高確信, $0.1 < \rho \leq 0.3$: 中確信, $\rho \leq 0.1$: 低確信）
    \item メタデータが利用可能な場合の商品名によるアイテム参照
\end{itemize}

\subsection{評価指標}

解釈品質を定量的に評価するため，以下の指標を定義する：

\begin{definition}[解釈品質指標]
\label{def:metrics}
解釈結果の集合$\mathcal{R} = \{r_1, \ldots, r_m\}$に対して：
\begin{itemize}
    \item 平均確信度: $\bar{c} = \frac{1}{m}\sum_{i=1}^{m} c_i$
    \item カバー率-確信度相関: $\text{corr}(\rho_i, c_i)$（ピアソン相関係数）
    \item 時間的記述完全率: $\frac{|\{r_i : |r_i.\text{temporal}| > 0\}|}{m}$
    \item 仮説生成完全率: $\frac{|\{r_i : |r_i.\text{hypothesis}| > 0\}|}{m}$
\end{itemize}
\end{definition}

\texttt{find\_dense\_itemsets}関数を密集パターン検出に使用した．

3つのウィンドウサイズ設定で実験を行った．

すべての設定において，時間的記述と仮説生成の完全率は100\%を達成した．

すべての要素を生成するように設計されているためであるが，
プロンプトテンプレートの構造的完全性を示す結果でもある．

ウィンドウサイズの増加に伴い検出パターン数が減少しているが（4$\to$2），
これは密集条件が厳しくなることによる自然な挙動である．

\subsection{解釈例}

以下に，Small設定で検出されたパターンの解釈例を示す：

\begin{quote}
\textbf{パターン}: $\{1, 2\}$ \\
\textbf{要約}: 「アイテム1とアイテム2は単一連続パターンを示し，
データ期間の100\%で密集的に共起しています．」 \\
\textbf{時間的特徴}: 連続した期間に集中しており，
安定した需要や継続的なプロモーション効果を示唆． \\
\textbf{仮説}: 定番商品の組み合わせ，または長期的なバンドル販売戦略の結果と推測．
\end{quote}

\subsection{メタデータ活用の効果}

商品名メタデータを付与した実験では，
「アイテム1」の代わりに「商品A」等の具体的な名称が
解釈文に反映され，可読性が大幅に向上した．
これは実運用において，アイテムIDマッピングテーブルの提供が
解釈品質に直結することを示唆している．

\section{議論}
\label{sec:discussion}

\subsection{LLM バックエンドへの拡張}

本論文ではモックバックエンドによるプロトタイプ検証を行ったが，
提案フレームワークのアーキテクチャは任意のLLM APIへの
接続を想定している．
GPT-4~\cite{openai2023gpt4}やLLaMA~\cite{touvron2023llama}等の
実際のLLMを接続した場合，以下の改善が期待される：
\begin{itemize}
    \item ドメイン知識に基づくより具体的なビジネス仮説の生成
    \item パターン間の暗黙的関係の発見
    \item 自然言語の表現の多様性と流暢性の向上
\end{itemize}

\subsection{限界と今後の課題}

本研究の主な限界は以下の通りである：
(1) モックバックエンドによる検証のため，実際のLLMの生成品質は未評価，
(2) 解釈の正確性の人間による評価が未実施，
(3) 大規模データセットでのスケーラビリティの検証が未実施．

今後の課題として，実LLMとの統合実験，
ドメイン専門家による解釈品質の人間評価，
および対話的な解釈精緻化インターフェースの設計が挙げられる．

\section{おわりに}
\label{sec:conclusion}

本論文では，密集アイテムセットパターンの自然言語解釈を
LLMにより自動生成するフレームワークを提案した．
パターン情報抽出・構造化プロンプト生成・品質評価指標の
3つのコンポーネントにより，解釈タスクを体系化した．

モックLLMバックエンドによるプロトタイプ実験では，
すべてのパターンに対して時間的記述とビジネス仮説を含む
解釈を生成できることを確認した．
32件のユニットテストにより実装の正確性を検証し，
フレームワークの堅牢性を示した．

提案フレームワークはバックエンドに非依存な設計であり，
実際のLLM APIとの統合により，データマイニング結果の
自動解釈ツールとしての実用化が期待される．
